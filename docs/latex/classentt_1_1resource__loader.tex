\subsubsection*{template$<$typename Loader, typename Resource$>$\newline
class entt\+::resource\+\_\+loader$<$ Loader, Resource $>$}

Base class for resource loaders. 

Resource loaders must inherit from this class and stay true to the C\+R\+TP idiom. Moreover, a resource loader must expose a public, const member function named {\ttfamily load} that accepts a variable number of arguments and returns a handle to the resource just created.~\newline
 As an example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{struct }my\_resource \{\};}
\DoxyCodeLine{}
\DoxyCodeLine{\textcolor{keyword}{struct }my\_loader: \mbox{\hyperlink{classentt_1_1resource__loader}{entt::resource\_loader}}<my\_loader, my\_resource> \{}
\DoxyCodeLine{    resource\_handle<my\_resource> load(\textcolor{keywordtype}{int} value)\textcolor{keyword}{ const }\{}
\DoxyCodeLine{        \textcolor{comment}{// use the integer value somehow}}
\DoxyCodeLine{        \textcolor{keywordflow}{return} std::make\_shared<my\_resource>();}
\DoxyCodeLine{    \}}
\DoxyCodeLine{\};}
\end{DoxyCode}


In general, resource loaders should not have a state or retain data of any type. They should let the cache manage their resources instead.

\begin{DoxyNote}{Note}
Base class and C\+R\+TP idiom aren\textquotesingle{}t strictly required with the current implementation. One could argue that a cache can easily work with loaders of any type. However, future changes won\textquotesingle{}t be breaking ones by forcing the use of a base class today and that\textquotesingle{}s why the model is already in its place.
\end{DoxyNote}
