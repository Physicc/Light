\subsubsection*{template$<$typename Entity$>$\newline
class entt\+::basic\+\_\+registry$<$ Entity $>$}

Fast and reliable entity-\/component system. 

The registry is the core class of the entity-\/component framework.~\newline
 It stores entities and arranges pools of components on a per request basis. By means of a registry, users can manage entities and components, then create views or groups to iterate them.

Returns a group for the given components. 

This kind of objects are created on the fly and share with the registry its internal data structures.~\newline
 Feel free to discard a group after the use. Creating and destroying a group is an incredibly cheap operation because they do not require any type of initialization, but for the first time they are requested.~\newline
 As a rule of thumb, storing a group should never be an option.

Groups support exclusion lists and can own types of components. The more types are owned by a group, the faster it is to iterate entities and components.~\newline
 However, groups also affect some features of the registry such as the creation and destruction of components, which will consequently be slightly slower (nothing that can be noticed in most cases).

\begin{DoxyNote}{Note}
Pools of components that are owned by a group cannot be sorted anymore. The group takes the ownership of the pools and arrange components so as to iterate them as fast as possible.
\end{DoxyNote}

Sorts two pools of components in the same way. 

The order of the elements in a pool is highly affected by assignments of components to entities and deletions. Components are arranged to maximize the performance during iterations and users should not make any assumption on the order.

It happens that different pools of components must be sorted the same way because of runtime and/or performance constraints. This function can be used to order a pool of components according to the order between the entities in another pool of components.

{\bfseries{How}} {\bfseries{it}} {\bfseries{works}} 

Being {\ttfamily A} and {\ttfamily B} the two sets where {\ttfamily B} is the master (the one the order of which rules) and {\ttfamily A} is the slave (the one to sort), after a call to this function an iterator for {\ttfamily A} will return the entities according to the following rules\+:


\begin{DoxyItemize}
\item All the entities in {\ttfamily A} that are also in {\ttfamily B} are returned first according to the order they have in {\ttfamily B}.
\item All the entities in {\ttfamily A} that are not in {\ttfamily B} are returned in no particular order after all the other entities.
\end{DoxyItemize}

Any subsequent change to {\ttfamily B} won\textquotesingle{}t affect the order in {\ttfamily A}.

Sorts the pool of entities for the given component. 

The order of the elements in a pool is highly affected by assignments of components to entities and deletions. Components are arranged to maximize the performance during iterations and users should not make any assumption on the order.~\newline
 This function can be used to impose an order to the elements in the pool of the given component. The order is kept valid until a component of the given type is assigned or removed from an entity.

The comparison function object must return {\ttfamily true} if the first element is {\itshape less} than the second one, {\ttfamily false} otherwise. The signature of the comparison function should be equivalent to one of the following\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{bool(\textcolor{keyword}{const} Entity, \textcolor{keyword}{const} Entity);}
\DoxyCodeLine{bool(\textcolor{keyword}{const} Component \&, \textcolor{keyword}{const} Component \&);}
\end{DoxyCode}


Moreover, the comparison function object shall induce a {\itshape strict weak ordering} on the values.

The sort function oject must offer a member function template {\ttfamily operator()} that accepts three arguments\+:


\begin{DoxyItemize}
\item An iterator to the first element of the range to sort.
\item An iterator past the last element of the range to sort.
\item A comparison function to use to compare the elements.
\end{DoxyItemize}

The comparison funtion object received by the sort function object hasn\textquotesingle{}t necessarily the type of the one passed along with the other parameters to this member function.

Returns a view for the given components. 

This kind of objects are created on the fly and share with the registry its internal data structures.~\newline
 Feel free to discard a view after the use. Creating and destroying a view is an incredibly cheap operation because they do not require any type of initialization.~\newline
 As a rule of thumb, storing a view should never be an option.

Views do their best to iterate the smallest set of candidate entities. In particular\+:


\begin{DoxyItemize}
\item Single component views are incredibly fast and iterate a packed array of entities, all of which has the given component.
\item Multi component views look at the number of entities available for each component and pick up a reference to the smallest set of candidates to test for the given components.
\end{DoxyItemize}

Views in no way affect the functionalities of the registry nor those of the underlying pools.

\begin{DoxyNote}{Note}
Multi component views are pretty fast. However their performance tend to degenerate when the number of components to iterate grows up and the most of the entities have all the given components.~\newline
 To get a performance boost, consider using a group instead.
\end{DoxyNote}
