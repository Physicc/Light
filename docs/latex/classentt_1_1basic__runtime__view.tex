\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename Entity$>$\newline
class entt\+::basic\+\_\+runtime\+\_\+view$<$ Entity $>$}

Runtime view. 

Runtime views iterate over those entities that have at least all the given components in their bags. During initialization, a runtime view looks at the number of entities available for each component and picks up a reference to the smallest set of candidate entities in order to get a performance boost when iterate.~\newline
 Order of elements during iterations are highly dependent on the order of the underlying data structures. See sparse\+\_\+set and its specializations for more details.

{\bfseries{Important}} 

Iterators aren\textquotesingle{}t invalidated if\+:


\begin{DoxyItemize}
\item New instances of the given components are created and assigned to entities.
\item The entity currently pointed is modified (as an example, if one of the given components is removed from the entity to which the iterator points).
\item The entity currently pointed is destroyed.
\end{DoxyItemize}

In all the other cases, modifying the pools of the given components in any way invalidates all the iterators and using them results in undefined behavior.

\begin{DoxyNote}{Note}
Views share references to the underlying data structures of the registry that generated them. Therefore any change to the entities and to the components made by means of the registry are immediately reflected by the views, unless a pool was missing when the view was built (in this case, the view won\textquotesingle{}t have a valid reference and won\textquotesingle{}t be updated accordingly).
\end{DoxyNote}
\begin{DoxyWarning}{Warning}
Lifetime of a view must not overcome that of the registry that generated it. In any other case, attempting to use a view results in undefined behavior.
\end{DoxyWarning}
