The \mbox{\hyperlink{group__window_ga3c96d80d363e67d13a41b5d1821f3242}{G\+L\+F\+Wwindow}} object encapsulates both a window and a context. They are created with \mbox{\hyperlink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}} and destroyed with \mbox{\hyperlink{group__window_ga806747476b7247d292be3711c323ea10}{glfw\+Destroy\+Window}}, or \mbox{\hyperlink{group__init_gafd90e6fd4819ea9e22e5e739519a6504}{glfw\+Terminate}}, if any remain. As the window and context are inseparably linked, the object pointer is used as both a context and window handle.

To see the event stream provided to the various window related callbacks, run the {\ttfamily events} test program.\hypertarget{window_guide_window_creation}{}\doxysubsection{Window creation}\label{window_guide_window_creation}
A window and its Open\+GL or Open\+GL ES context are created with \mbox{\hyperlink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}}, which returns a handle to the created window object. For example, this creates a 640 by 480 windowed mode window\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{group__window_ga3c96d80d363e67d13a41b5d1821f3242}{GLFWwindow}}* window = \mbox{\hyperlink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfwCreateWindow}}(640, 480, \textcolor{stringliteral}{"My Title"}, NULL, NULL);}
\end{DoxyCode}


If window creation fails, {\ttfamily N\+U\+LL} will be returned, so it is necessary to check the return value.

The window handle is passed to all window related functions and is provided to along with all input events, so event handlers can tell which window received the event.\hypertarget{window_guide_window_full_screen}{}\doxysubsubsection{Full screen windows}\label{window_guide_window_full_screen}
To create a full screen window, you need to specify which monitor the window should use. In most cases, the user\textquotesingle{}s primary monitor is a good choice. For more information about retrieving monitors, see \mbox{\hyperlink{monitor_guide_monitor_monitors}{Retrieving monitors}}.

\end{DoxyCode}


Full screen windows cover the entire display area of a monitor, have no border or decorations.

Windowed mode windows can be made full screen by setting a monitor with \mbox{\hyperlink{group__window_ga12fabf78575e59c00f822f323ae0b6ae}{glfw\+Set\+Window\+Monitor}}, and full screen ones can be made windowed by unsetting it with the same function.

Each field of the \mbox{\hyperlink{structGLFWvidmode}{G\+L\+F\+Wvidmode}} structure corresponds to a function parameter or window hint and combine to form the {\itshape desired video mode} for that window. The supported video mode most closely matching the desired video mode will be set for the chosen monitor as long as the window has input focus. For more information about retrieving video modes, see \mbox{\hyperlink{monitor_guide_monitor_modes}{Video modes}}.

Once you have a full screen window, you can change its resolution, refresh rate and monitor with \mbox{\hyperlink{group__window_ga12fabf78575e59c00f822f323ae0b6ae}{glfw\+Set\+Window\+Monitor}}. If you only need change its resolution you can also call \mbox{\hyperlink{group__window_gae54d1f4915ded15e267ddd3f41496cd2}{glfw\+Set\+Window\+Size}}. In all cases, the new video mode will be selected the same way as the video mode chosen by \mbox{\hyperlink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}}. If the window has an Open\+GL or Open\+GL ES context, it will be unaffected.

By default, the original video mode of the monitor will be restored and the window iconified if it loses input focus, to allow the user to switch back to the desktop. This behavior can be disabled with the \mbox{\hyperlink{window_guide_GLFW_AUTO_ICONIFY_hint}{G\+L\+F\+W\+\_\+\+A\+U\+T\+O\+\_\+\+I\+C\+O\+N\+I\+FY}} window hint, for example if you wish to simultaneously cover multiple monitors with full screen windows.

If a monitor is disconnected, all windows that are full screen on that monitor will be switched to windowed mode. See \mbox{\hyperlink{monitor_guide_monitor_event}{Monitor configuration changes}} for more information.\hypertarget{window_guide_window_windowed_full_screen}{}\doxysubsubsection{\char`\"{}\+Windowed full screen\char`\"{} windows}\label{window_guide_window_windowed_full_screen}
If the closest match for the desired video mode is the current one, the video mode will not be changed, making window creation faster and application switching much smoother. This is sometimes called {\itshape windowed full screen} or {\itshape borderless full screen} window and counts as a full screen window. To create such a window, request the current video mode.

Window destruction always succeeds. Before the actual destruction, all callbacks are removed so no further events will be delivered for the window. All windows remaining when \mbox{\hyperlink{group__init_gafd90e6fd4819ea9e22e5e739519a6504}{glfw\+Terminate}} is called are destroyed as well.

When a full screen window is destroyed, the original video mode of its monitor is restored, but the gamma ramp is left untouched.\hypertarget{window_guide_window_hints}{}\doxysubsection{Window creation hints}\label{window_guide_window_hints}
There are a number of hints that can be set before the creation of a window and context. Some affect the window itself, others affect the framebuffer or context. These hints are set to their default values each time the library is initialized with \mbox{\hyperlink{group__init_gab41771f0215a2e0afb4cf1cf98082d40}{glfw\+Init}}. Integer value hints can be set individually with \mbox{\hyperlink{group__window_ga69c40728499720bef8a49aa925ea0efa}{glfw\+Window\+Hint}} and string value hints with \mbox{\hyperlink{group__window_ga3c2270562d7ba13d92680ba1090adafe}{glfw\+Window\+Hint\+String}}. You can reset all at once to their defaults with \mbox{\hyperlink{group__window_ga8050ddceed9dc6bd9d3aa35666195cd4}{glfw\+Default\+Window\+Hints}}.

Some hints are platform specific. These are always valid to set on any platform but they will only affect their specific platform. Other platforms will ignore them. Setting these hints requires no platform specific headers or calls.

\begin{DoxyNote}{Note}
Window hints need to be set before the creation of the window and context you wish to have the specified attributes. They function as additional arguments to \mbox{\hyperlink{group__window_gaaccd00fafe81e2db94e8cdf55721e055}{glfw\+Create\+Window}}.
\end{DoxyNote}
\hypertarget{window_guide_window_hints_hard}{}\doxysubsubsection{Hard and soft constraints}\label{window_guide_window_hints_hard}
Some window hints are hard constraints. These must match the available capabilities {\itshape exactly} for window and context creation to succeed. Hints that are not hard constraints are matched as closely as possible, but the resulting context and framebuffer may differ from what these hints requested.

{\bfseries{Open\+GL\+:}} If enabled and supported by the system, the {\ttfamily G\+L\+\_\+\+F\+R\+A\+M\+E\+B\+U\+F\+F\+E\+R\+\_\+\+S\+R\+GB} enable will control s\+R\+GB rendering. By default, s\+R\+GB rendering will be disabled.
\end{DoxyParagraph}
\begin{DoxyParagraph}{}
{\bfseries{Open\+GL ES\+:}} If enabled and supported by the system, the context will always have s\+R\+GB rendering enabled.
\end{DoxyParagraph}

\label{window_guide_GLFW_CONTEXT_CREATION_API_hint}%

{\bfseries{G\+L\+F\+W\+\_\+\+C\+O\+N\+T\+E\+X\+T\+\_\+\+C\+R\+E\+A\+T\+I\+O\+N\+\_\+\+A\+PI}} specifies which context creation A\+PI to use to create the context. Possible values are {\ttfamily G\+L\+F\+W\+\_\+\+N\+A\+T\+I\+V\+E\+\_\+\+C\+O\+N\+T\+E\+X\+T\+\_\+\+A\+PI}, {\ttfamily G\+L\+F\+W\+\_\+\+E\+G\+L\+\_\+\+C\+O\+N\+T\+E\+X\+T\+\_\+\+A\+PI} and {\ttfamily G\+L\+F\+W\+\_\+\+O\+S\+M\+E\+S\+A\+\_\+\+C\+O\+N\+T\+E\+X\+T\+\_\+\+A\+PI}. This is a hard constraint. If no client A\+PI is requested, this hint is ignored.

\begin{DoxyParagraph}{}
@macos The E\+GL A\+PI is not available on this platform and requests to use it will fail.
\end{DoxyParagraph}
\begin{DoxyParagraph}{}

\end{DoxyParagraph}

{\bfseries{G\+L\+F\+W\+\_\+\+C\+O\+N\+T\+E\+X\+T\+\_\+\+V\+E\+R\+S\+I\+O\+N\+\_\+\+M\+A\+J\+OR}} and {\bfseries{G\+L\+F\+W\+\_\+\+C\+O\+N\+T\+E\+X\+T\+\_\+\+V\+E\+R\+S\+I\+O\+N\+\_\+\+M\+I\+N\+OR}} specify the client A\+PI version that the created context must be compatible with. The exact behavior of these hints depend on the requested client A\+PI.

\begin{DoxyNote}{Note}
Do not confuse these hints with {\ttfamily G\+L\+F\+W\+\_\+\+V\+E\+R\+S\+I\+O\+N\+\_\+\+M\+A\+J\+OR} and {\ttfamily G\+L\+F\+W\+\_\+\+V\+E\+R\+S\+I\+O\+N\+\_\+\+M\+I\+N\+OR}, which provide the A\+PI version of the G\+L\+FW header.
\end{DoxyNote}
\begin{DoxyParagraph}{}
{\bfseries{Open\+GL\+:}} These hints are not hard constraints, but creation will fail if the Open\+GL version of the created context is less than the one requested. It is therefore perfectly safe to use the default of version 1.\+0 for legacy code and you will still get backwards-\/compatible contexts of version 3.\+0 and above when available.
\end{DoxyParagraph}
\begin{DoxyParagraph}{}
While there is no way to ask the driver for a context of the highest supported version, G\+L\+FW will attempt to provide this when you ask for a version 1.\+0 context, which is the default for these hints.
\end{DoxyParagraph}
\begin{DoxyParagraph}{}
{\bfseries{Open\+GL ES\+:}} These hints are not hard constraints, but creation will fail if the Open\+GL ES version of the created context is less than the one requested. Additionally, Open\+GL ES 1.\+x cannot be returned if 2.\+0 or later was requested, and vice versa. This is because Open\+GL ES 3.\+x is backward compatible with 2.\+0, but Open\+GL ES 2.\+0 is not backward compatible with 1.\+x.
\end{DoxyParagraph}
\begin{DoxyNote}{Note}
@macos The OS only supports forward-\/compatible core profile contexts for Open\+GL versions 3.\+2 and later. Before creating an Open\+GL context of version 3.\+2 or later you must set the \mbox{\hyperlink{window_guide_GLFW_OPENGL_FORWARD_COMPAT_hint}{G\+L\+F\+W\+\_\+\+O\+P\+E\+N\+G\+L\+\_\+\+F\+O\+R\+W\+A\+R\+D\+\_\+\+C\+O\+M\+P\+AT}} and \mbox{\hyperlink{window_guide_GLFW_OPENGL_PROFILE_hint}{G\+L\+F\+W\+\_\+\+O\+P\+E\+N\+G\+L\+\_\+\+P\+R\+O\+F\+I\+LE}} hints accordingly. Open\+GL 3.\+0 and 3.\+1 contexts are not supported at all on mac\+OS.
\end{DoxyNote}

When calling {\ttfamily gl\+Get\+Framebuffer\+Attachment\+Parameteriv}, the red, green, blue and alpha sizes are queried from the {\ttfamily G\+L\+\_\+\+B\+A\+C\+K\+\_\+\+L\+E\+FT}, while the depth and stencil sizes are queried from the {\ttfamily G\+L\+\_\+\+D\+E\+P\+TH} and {\ttfamily G\+L\+\_\+\+S\+T\+E\+N\+C\+IL} attachments, respectively.\hypertarget{window_guide_buffer_swap}{}\doxysection{Buffer swapping}\label{window_guide_buffer_swap}
G\+L\+FW windows are by default double buffered. That means that you have two rendering buffers; a front buffer and a back buffer. The front buffer is the one being displayed and the back buffer the one you render to.

When the entire frame has been rendered, it is time to swap the back and the front buffers in order to display what has been rendered and begin rendering a new frame. This is done with \mbox{\hyperlink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfw\+Swap\+Buffers}}.


\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfwSwapBuffers}}(window);}
\end{DoxyCode}


Sometimes it can be useful to select when the buffer swap will occur. With the function \mbox{\hyperlink{group__context_ga12a595c06947cec4967c6e1f14210a8a}{glfw\+Swap\+Interval}} it is possible to select the minimum number of monitor refreshes the driver should wait from the time \mbox{\hyperlink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfw\+Swap\+Buffers}} was called before swapping the buffers\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{group__context_ga12a595c06947cec4967c6e1f14210a8a}{glfwSwapInterval}}(1);}
\end{DoxyCode}


If the interval is zero, the swap will take place immediately when \mbox{\hyperlink{group__window_gafb827800eedbfcbc97b1e5408df668d7}{glfw\+Swap\+Buffers}} is called without waiting for a refresh. Otherwise at least interval retraces will pass between each buffer swap. Using a swap interval of zero can be useful for benchmarking purposes, when it is not desirable to measure the time it takes to wait for the vertical retrace. However, a swap interval of one lets you avoid tearing.

Note that this may not work on all machines, as some drivers have user-\/controlled settings that override any swap interval the application requests.

A context that supports either the {\ttfamily W\+G\+L\+\_\+\+E\+X\+T\+\_\+swap\+\_\+control\+\_\+tear} or the {\ttfamily G\+L\+X\+\_\+\+E\+X\+T\+\_\+swap\+\_\+control\+\_\+tear} extension also accepts {\itshape negative} swap intervals, which allows the driver to swap immediately even if a frame arrives a little bit late. This trades the risk of visible tears for greater framerate stability. You can check for these extensions with \mbox{\hyperlink{group__context_ga9a28c712d35f9e43534e1d03b051c04c}{glfw\+Extension\+Supported}}.
